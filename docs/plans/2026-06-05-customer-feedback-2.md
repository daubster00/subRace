### 2026-06-05 — 아키텍처 방향 합의: 사전 스케줄 + 시간 단위

**전제 정리** (이걸 흐리지 말 것):
- 이 프로젝트는 **API 폴링 사이의 빈 구간을 가짜 움직임으로 채우는 것이 기본 작동 원리**.
- 70%의 채널이 YouTube API 반올림 정체 상태인 건 처음부터 알고 있던 상수, 버그 아님.
- 고객 피드백은 전부 "가짜 움직임이 진짜 데이터 흐름과 닮지 않은 부분" 지적. "가짜 멈춰주세요"가 아님.
- → "데이터 신뢰도 낮으니 sit-still" 같은 발상은 프로젝트 가치 부정. 모든 채널은 항상 움직인다.

**합의된 아키텍처 방향:**

1. **현행 랜덤 동전 던지기 방식 폐기, 사전 스케줄 방식으로 전환.**
   - 현재 `decideDirection` / `pickStepMagnitude` / `decideChangeCount` 의 "매 이벤트마다 랜덤" 구조 폐기.
   - 채널별로 시간 단위 이벤트 스케줄(시각·방향·크기)을 미리 다 짜놓고 그대로 실행.

2. **단위는 1시간** (현행 하루 단위 X).
   - 하루치를 한 번에 정해버리면 mid-day API 점프 catch-up 불가가 고객 항목 1의 진짜 원인.
   - 시간 단위면 최악 1시간치 어긋남 → 다음 시간에 재할당해서 자연 흡수.
   - (사용자 메모: 15분 단위로 더 잘게 쪼개는 안도 추후 검토 가능.)

3. **재스케줄 트리거 2종**:
   - **정각 tick**: 모든 채널의 다음 1시간 스케줄 새로 짬 (현재 pace × 1h 기준).
   - **API 변동**: 영향받은 채널만, 그 시간 남은 슬롯·magnitude 재배분 + 필요 시 이벤트 추가.

4. **이벤트 내용은 사전 확정 (랜덤 X)**:
   - **시각**: 평균 간격(예: 120초)에서 ±jitter (예: ±30초). 메트로놈처럼 안 보이게.
   - **크기**: 남은 양 ÷ 남은 이벤트 수, 거기에 ±10% jitter. 매번 stepBounds 안 랜덤이 아님.
     - 이 변경으로 항목 4 (40~50씩 튐) 자연 해소: 큰 튐 자체가 안 나옴.
   - **방향**: 우세 방향 이벤트 수와 반대 방향 이벤트 수를 **개수로 확정** (확률 X). 예: 시간당 30 이벤트 → 24 up + 6 down.
     - 6개 down 이벤트를 1시간 안에 **고르게 분산**해서 박음 (한 군데 몰리지 않게).
     - 이 변경으로 항목 6 (1시간 단위로 보면 감소처럼 보임) 자연 해소: 우연한 몰림이 구조적으로 불가능.

5. **방향 분포 비율은 추후 결정**.
   - 출발 제안 80/20이지만, 항목 6 고려해 90/10으로 더 보수적으로 갈 수도. 데이터 보고 정함.

6. **Catch-up 로직 명시**:
   - API가 시간 중간에 점프해 추정치를 추월하면, 그 채널의 시간 남은 슬롯 수·크기를 재배분해서 따라잡음.
   - 현재 step·count 상한 (sub-10M: max +30/event, max 720/day)이 catch-up 시나리오 발목 잡으면 풀어야 함.
   - 이 변경으로 항목 1, 11 자연 해소.

**합의된 트레이드오프 인식:**
- 사전 스케줄이 자연스럽지 못할 거란 통념은 자연스러움이 잘못 정의된 경우. 자연스러움은 한 채널 내부의 흐름(jitter)에서 나오지 채널 뽑기의 무작위성에서 나오지 않음.
- 진짜 비용은 (a) 스케줄 테이블 추가 마이그레이션, (b) 재스케줄 빈도에 따른 DB 쓰기 부담, (c) 첫 적용 후 jitter 파라미터 튜닝 필요.

**이 방향으로 해결되는 고객 항목**: 1, 4, 6, 11 (구조적 해결). 5도 일부 흡수.

### 2026-06-05 (속행) — 스케줄링 4규칙 + 추가 결정

**스케줄링 핵심 4규칙:**

1. **새 마일스톤 강제 catch-up** — 기존과 다른 새 마일스톤 값이 들어오면 **반드시 1시간 안에** 그 마일스톤 값에 도달.
2. **Target 추종 단계** — 마일스톤 값 도달 후엔 target 값에 맞춰 스케줄링.
3. **시간당 목표 + 이벤트 간격** — **최소 6개, 최대 8개의 최근 마일스톤을 참고**해서 **추세를 읽고** 시간당 목표 변화량 산출. (간격이 점점 벌어지는지/줄어드는지 등). **최근 가중치(recent-weighted)** 적용. 사용자가 모든 채널에 최소 6개 마일스톤 수동 시드 보장 → fallback 불필요.
4. **Target 공식** — 새 마일스톤과 직전 마일스톤의 부호(±)로 방향 결정 후, **다음 예측 마일스톤의 95%** 로 target 설정.
   - 공식: `target = 새 마일스톤 + 0.95 × (새 마일스톤 − 직전 마일스톤)`
   - 예시: 5,680k → 5,690k → target = 5,699,500
   - 하락 예시: 626,700 위치에서 새 마일스톤 625k(직전 626k) 들어오면 → target = 624,050

**추가 결정사항:**

- **Target 도달 후 변동 폭 (규칙 2 보완)**: ±**3% of step**. 
  - 예: step=10k → ±300 사이에서 증감 반복.
  - target에 도달하면 그 폭 안에서 계속 흔들리고 있다가 새 마일스톤 도래 시 규칙 1로 점프.
- **80/20 방향 비율의 정확한 의미**: "증가 80%/감소 20%"가 아니라 **트렌드 방향 80%, 반대 방향 20%**.
  - 채널이 상승 트렌드면: 80% up + 20% down
  - 채널이 하락 트렌드면: 80% down + 20% up
- **Catch-up 중에는 100% 트렌드 방향** (반대 방향 이벤트 0). catch-up 끝나면 80/20으로 복귀.
- **이벤트 단위 magnitude 상한 = 20** — 한 번 증감 시 절대 ±20을 넘지 않음.
- **휴식 구간 필수** — 1시간 스케줄 안에 모든 채널이 동시에 계속 움직이지 않도록, **이벤트 사이 휴식 시간** 포함. 휴식 분포 자체가 채널 간 동시 움직임을 자연 분산시키는 장치 (랜덤 샘플러 대체).

**주의 (사용자 메모):**
- 폴링 주기 2분 → 한 폴링 사이 api 변화는 **반드시 ±1 마일스톤 step 이하** (반올림 구조). 한 번에 여러 마일스톤 점프 같은 상황은 **절대 없음**. catch-up 분량은 항상 1 step 이내.
- 기존 코드의 step bounds(30) / change count(720) 같은 한도값은 **완전히 무시**. 새 시스템은 백지에서 산출 ([[feedback_fresh_design]]).

- **Worker 재시작 시** = in-flight 스케줄 복구 안 함, 즉시 재계획 (단순함 우선)

**스케줄 사이클 (중요 정정)**:
- 정각 단위 글로벌 tick 아님 ❌
- **채널마다 자체 1시간 사이클**: 마지막 리셋 시점(import / catch-up / target 도달 / 새 마일스톤 등)으로부터 1시간 후가 다음 리셋
- 채널별 리셋 시각이 자연스럽게 분산 → 모든 채널이 동시에 재스케줄 X → 부하 분산 + 자연 분포
- 시간이 흐를수록 채널 간 리셋 타임이 더 발산

**사이클 구현 세부 (확정)**:
- **Worker tick = 1분**. 매분 "지금 시각 > next_cycle_reset_at 인 채널" 찾아서 재스케줄.
- **DB 필드**: `display_state.next_cycle_reset_at` (또는 별도 스케줄 테이블에) 추가.
- **다음 리셋 시각 계산**:
  - 시간 만료 경로: `마지막 리셋 시각 + 1시간`
  - Catch-up 경로: `catch-up 완료 시각 + 1시간`
- 두 경로 중 하나가 트리거되면 다른 쪽 시계 reset.

**시간당 최소 이벤트 수 (확정)**: **N = 6** (평균 10분 간격). 시간당 목표가 작은 느린 채널도 시간 안에 6번은 움직이도록 강제. 80/20 비율로 분배 후 magnitude 조정해서 net 합산이 시간당 목표와 일치.

**항목 2 (마일스톤 dedup) fix — 코드 작업 전에 선행**:
- **사용자 원안 정정**: dedup의 본래 의도는 "직전 마일스톤과 같은 값이면 skip" (시간 차원 상관없음). 같은 값이라도 시간 두고 다시 도래하면 새 row로 기록되어야 함.
- 예: 4월 14.1M → 5월 14.2M → 6월 14.1M = 6월 14.1M도 새 row로 기록.
- 현재 worker 로직(`worker/youtube-channels.ts:151`)은 이미 poll_state로 직전 값 비교해서 같으면 skip 함. **올바름.**
- **막고 있는 건 `subscriber_snapshots`의 UNIQUE INDEX `(channel_id, subscriber_count)` 하나뿐** (`migrations/007_milestone_history.sql:43-44`).
- **Fix = 그 인덱스 DROP**. 또는 `(channel_id, subscriber_count, polled_at)` 으로 변경 (다중 컬럼 unique). 후자가 동시성 중복 방어 유지하면서도 시간 다른 같은 값 허용. **후자 추천.**

**DB 스키마 신설**: 이벤트 스케줄 테이블 1개 추가 (channel_id, scheduled_at, magnitude, direction, applied flag). 시간 단위 결정(1시간)에 맞춰 설계.

### 2026-06-05 (속행 3) — A~E 답변 + 정정사항

**A. Target 도달 후 진동 방향** = bouncing 채택. 위 경계 닿으면 down, 아래 닿으면 up.

**B. 느린 채널 동작 정정**: "시간당 < 1 이벤트면 통째 휴식" ❌. 느린 채널도 **시간 안에 증감 반복하면서 시간당 목표 도달**해야 함. 빈도만 낮고 멈추진 않음.
- 예: 시간당 목표 +14, 이벤트 4개 → +10, -3, +5, +2 = +14 (3 up + 1 down).
- 최소 이벤트 수 보장 필요 (예: 시간당 최소 N개). N 값은 추후 결정.

**C. 마일스톤 시간 간격 산출 방식** = **최근 가중 평균** (단순 평균 X). 최신 간격에 weight 더 높게.

**D. 마일스톤 수동 import 후 첫 스케줄** = **import 즉시** (정각·자정 대기 X). 테스트 빨리 돌리고 안 되면 바로 수정.

**E. Playboard 데이터 — 사용자 결정 필요. 별도 설명 참조** (아래).

**규칙 3 대폭 갱신 (사용자 지시)**:
- "최근 3개 마일스톤 평균" → **6~8개 마일스톤으로 추세 읽기** (사용자가 모든 채널 6개 최소 시드 보장)

**추세 분석 알고리즘 — 확정 (2026-06-05 협의)**:

**중요 — 규칙 3·4의 관계 정정**:
- 규칙 3은 별도의 "시간당 변화량"을 산출하는 게 아님
- 규칙 4가 destination(target 값)을 잡고, 규칙 3은 **그 target까지 예상 도달 시간**을 산출
- **시간당 목표값 = (target − 현재 display) ÷ 예상 도달 시간(시간)**
- 즉 규칙 3은 "다음 마일스톤이 얼마나 뒤에 도착할지" 예측하는 것

**예상 도달 시간 산출 = 최근 마일스톤 간격의 순서 기반 선형 가중 평균**
- 최근 6~8개 마일스톤의 인접 간격(시간)에 순서별 weight 적용
- weight: 가장 최근 = 8, 그 다음 = 7, ..., 가장 옛것 = 1
  - 예: 간격 8개일 때 `Σ(간격 × weight) / 36 (= 8+7+...+1)`
  - 간격 6개일 때 weight는 6,5,4,3,2,1 → 합 21
- 날짜 무관, 순서만 사용 (λ 같은 튜닝 변수 없음)
- 가속/감속이 자연스럽게 가중치에 녹아 들어감 — **별도 ×1.1/×0.9 보정 불필요**
- 부호(±) 신호는 규칙 4가 즉시 반응적으로 처리, 추세 분석에서 별도로 다루지 않음

**시간당 목표값 산출 흐름**:
1. 새 마일스톤 도착 → 규칙 4로 target 산출
2. 규칙 3로 다음 마일스톤까지 예상 시간 산출 (시간 단위)
3. 시간당 목표값 = (target − 현재 display) / 예상 시간
4. 1시간 사이클마다 위 계산 재실행 (마일스톤 안 들어와도, 도달 시간 추정 변동 반영)
