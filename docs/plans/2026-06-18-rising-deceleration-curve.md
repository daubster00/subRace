# 상승 채널 감속 곡선 (0.9배 속도 + 90/93/97/99% 단계 감속)

**작성:** 2026-06-18
**범위:** 상승 추세 채널이 천장에 다가가는 "속도 곡선"만. 하락·정체 채널
(2026-06-18 천장 준수 수정)과 이벤트 생성·SSE 배관은 손대지 않는다.

---

## 배경

고객 1차 요청(오해 정정본):
- 추정치는 예상 속도의 **0.9 수준**으로 이동.
- 단, **다음 API 마일스톤 한 칸 구간의 90% 지점부터** 점진 감속하여 실제 API 값을
  추월하지 않게.
- 90%에서 멈추는 게 아니라 90~93%, 93~97%, 97~99%로 구간별 감속.
- 97~99% 부근에서 다음 마일스톤을 **절대 추월 금지**.

현재 상승 분기는 매 사이클 "남은 거리 × (사이클 시간 ÷ 다음 마일스톤 예상 시간)"
으로 이동한다. 거리비례라 자연 감속은 되지만 0.9배 속도·단계 감속이 명시적으로
박혀 있지 않다.

## 확정 결정 (2026-06-18)

- **퍼센트 기준 = 한 칸 전체** `[마일스톤(=현재 API값) → 다음 마일스톤(한 단위 위)]`.
- **최종 주차 = 99%.** 다음 마일스톤(100%)은 절대 안 넘는다.
- 따라서 **상승 채널 천장이 95% → 99%로 상향**된다(직전 배포 0.95는 상승 한정
  대체). 하락·정체 채널의 "마일스톤+1%" 밴드는 그대로 유지.
- 한 칸 1만 단위 예시: 55,100,000 → 감속 시작 55,109,000(90%), 주차 55,109,900(99%).

## 모델 (매 사이클, 상승 분기에서만)

1. **위치 비율** `p = (화면값 − 마일스톤) / 한 단위` (0~1).
2. **예상 속도** = 한 단위 ÷ 예상 도달 시간(최근 마일스톤 간격 가중평균, 이미 산출 중).
3. **기본 속도** = 예상 속도 × **0.9**.
4. **단계 감속 배수** `g(p)`:

   | 위치 비율 p | 속도 배수 |
   |---|---|
   | p < 0.90 | 1.0 |
   | 0.90 ≤ p < 0.93 | 0.5 |
   | 0.93 ≤ p < 0.97 | 0.25 |
   | 0.97 ≤ p < 0.99 | 0.1 |
   | p ≥ 0.99 | 0 (주차 — target-bounce 미세 진동) |

   ※ 배수 값은 출발 제안. 적용 후 화면 보고 튜닝.

5. **이번 사이클 이동량** = 기본 속도 × g(p) × 사이클 시간.
   단 **화면값 + 이동량 ≤ 마일스톤 + 0.99 × 단위**(추월 금지 하드 클램프).

사이클마다 p를 다시 재므로 위로 갈수록 배수가 작아져 99% 천장에 매끄럽게 수렴한다
(단계 감속이 사이클 재계획에 자연스럽게 녹음).

## 천장 도달·새 마일스톤 인입 시

- 화면값이 99%에 주차 → 다음 마일스톤이 API로 확정되면 마일스톤이 한 칸 위로 점프,
  화면값(옛 99% = 새 마일스톤 −1%)은 새 칸 바닥 바로 아래라 그대로 이어서 상승.
  **상승 catch-up 불필요**(사용자 직관대로) — 격차가 한 칸의 1%뿐.
- 예상보다 빨라 API가 먼저 점프하면: 마일스톤 점프로 화면값이 새 바닥 밑에 놓여
  상승 페이스가 크게 잡혀 빠르게 따라붙음(runaway lag 없음).

## 검토할 열린 항목 (구현 단계에서 확정)

1. **0.9배 기본 속도와 거리비례식의 관계** — 기존 "남은 거리 × 시간비"를 버리고
   "0.9 × 예상속도 × g(p)"로 완전 교체할지, 둘을 결합할지. 권장: 완전 교체(명시적·튜닝 쉬움).
2. **99% 주차 시 진동 phase 진입** — p ≥ 0.99면 target-bounce로 보내고 진폭은
   작게(예: 한 단위의 0.1~0.3%). 기존 CF-17 "마일스톤+1% catch-up" 경로는 이 모델에선
   불필요 가능 → 제거 검토.
3. **느린 채널** — g(p) 적용 후 사이클 이동량이 매우 작아져도 멈추지 않고 최소 이벤트
   수는 보장(기존 buildCycleEvents의 N_MIN 로직 재사용).
4. **감속 배수 5단계 값** — 위 표는 출발값. 시뮬레이션 + 실화면으로 조정.

## 손대는 곳

- `src/lib/schedule-plan.ts` — 상승 분기 페이스 산출부 교체. 하락·정체 분기·이벤트
  생성·배관 무변경.
- `src/lib/milestone-delta.ts` — 상승 target/ceiling 비율 0.95 → 0.99(또는 별도
  ceilingRatio 신설). trendSign·예상시간 산출 로직은 유지.
- env: 감속 밴드 경계(0.90/0.93/0.97/0.99)·배수·기본속도계수(0.9)를 설정값으로.
- 테스트: 상승 페이스 단위 테스트 갱신 + 밴드별 감속·99% 추월 금지 케이스 추가.

## 검증 계획

1. 단위 테스트: p가 각 밴드일 때 이동량이 표대로 줄고, 어떤 경우도 99% 천장을
   안 넘는지.
2. 시뮬레이션: 한 상승 채널을 여러 사이클 돌려 화면값 궤적이 90%부터 단계적으로
   느려져 99%에 수렴, 100% 절대 미도달 확인. + 예상보다 빠른/느린 채널에서
   runaway lag 없음 확인.
3. 로컬 빌드·재기동 후 실데이터 일부 상승 채널로 한 사이클 관측.
4. 서버 반영 → 강제 재계획 → 천장 초과 0 유지 + 상승 채널이 99%에 붙는지 확인.
