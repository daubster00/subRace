# SubRace (라이브서브랭크) — PRD

**버전**: 0.1 (초안)
**작성일**: 2026-05-07
**상위 입력**: `docs/requirements/brief.md`
**범위**: 단일 페이지 운영자 전용 대시보드 (OBS 송출용)

---

## 1. 제품 개요

### 1.1 목적
일본 유튜브 채널 구독자 수 순위를 **실시간에 가깝게** 보여주는 단일 페이지 대시보드. 운영자가 OBS로 화면을 캡처하여 라이브 방송에 송출하는 용도로 사용한다.

### 1.2 사용자
- **단일 사용자**: 운영자 1명
- **간접 사용자**: 방송 시청자 (OBS 송출 화면을 일방향으로 시청)
- **인터랙션**: 없음. 운영자도 페이지를 띄워두기만 하며, 화면을 클릭/조작하지 않는다.

### 1.3 사용 시나리오
1. 운영자는 방송 전 페이지에 접속한다 (Basic Auth로 인증).
2. OBS의 브라우저 소스 또는 윈도우 캡처로 페이지를 송출원으로 등록한다.
3. 방송 동안 화면은 자동으로 데이터를 갱신하고 애니메이션을 표시한다.
4. API 장애 시에도 화면은 마지막 정상 데이터를 유지하며 송출을 방해하지 않는다.

## 3. 화면 구조 (컴포넌트 인벤토리)

디자인 시안 `자료/SubRace_design_50.png` 기준. 작동 프로토타입은 `자료/live_subrank_coding/` 참조.

```
┌─────────────────────────────────────────────────────────────────┐
│ [Header]                                                        │
│   ▶ 라이브서브랭크 (브랜드 마크 + 명)                            │
├─────────────────────────────────────────────────────────────────┤
│ [Hero]                                                          │
│   제목: 실시간 유튜브 구독자 순위 TOP 50                         │
│   설명: 실시간으로 집계되는 유튜브 채널 구독자 순위입니다.        │
│   [SummaryCard×3]: 시청 중 / 좋아요 / 업데이트                   │
├─────────────────────────────────────────────────────────────────┤
│ [Legend]                                                        │
│   ● 구독자 증가  ● 구독자 감소  ● 순위 변동 임박                 │
├─────────────────────────────────────────────────────────────────┤
│ [RankGrid]                                                      │
│   5개 컬럼 (1~10, 11~20, 21~30, 31~40, 41~50)                   │
│   각 컬럼은 10개의 RankCard 세로 배치                            │
├─────────────────────────────────────────────────────────────────┤
│ [BottomPanels]                                                  │
│   [LiveStatusPanel] [NextUpdatePanel] [RankAlertPanel]          │
│                                            [Timestamp (JST)]    │
└─────────────────────────────────────────────────────────────────┘
```

### 3.1 SummaryCard (상단 3장)
- **시청 중**: **클라이언트 채널의 현재 라이브 동시 시청자 수**. YouTube Data API v3 (`videos.list?part=liveStreamingDetails`의 `concurrentViewers`).
- **좋아요**: **클라이언트 채널 최신 또는 라이브 영상의 좋아요 수**. YouTube Data API v3 (`videos.list?part=statistics`의 `likeCount`).
- **업데이트**: "마지막으로 화면 데이터가 갱신된 시점"을 상대 시간으로 표시 (예: "0.8초 전")

> *상태*: 클라이언트 채널이 아직 없음. 개발 중에는 임의의 다른 채널을 사용하고, 채널 ID는 환경변수 `CLIENT_CHANNEL_ID`로 추출하여 추후 교체 용이하게 처리.
> *결정 필요(소)*: 라이브 스트림이 없을 때 "시청 중"의 동작 — 0 표시 / 카드 숨김 / 마지막 라이브 값 표시 중 어느 것? (구현 시 결정, 기본은 0)

### 3.2 RankCard (각 채널)
- 순위 번호
- 채널 로고 (그라데이션 또는 채널 썸네일)
- 채널명
- 구독자 수 (롤링 카운터로 표시)
- 증감 표시 (상승/하락 아이콘 + 델타 값, 초록/빨강)
- 순위 변동 임박 시 카드 자체 강조 (빨간 배경/테두리/글로우)

### 3.3 BottomPanels
- **LiveStatusPanel**: "실시간 상태" — 점(파란색/시안) + 웨이브 그래프. **장애 시 점/문구 색상만 경고색으로 변경**.
- **NextUpdatePanel**: 다음 YouTube API 폴링까지 남은 시간 (`mm:ss` 또는 "n초 후 ...").
- **RankAlertPanel**: 순위 변동 임박인 채널 쌍의 개수 (예: "2"). 빨간 벨 아이콘 + 숫자 배지.
- **Timestamp**: `2025.05.24 14:30:00 (JST)` 형식. 마지막 데이터 기준 시각.

## 4. 기능 요구사항 (Acceptance Criteria 중심)

### 4.1 데이터 파이프라인

#### F-1. Social Blade 채널 목록 폴링
- **주기**: 48시간마다 1회
- **대상**: 일본 YouTube 상위 150개 채널 목록 (Top List)
- **저장**: 채널 ID, 핸들, 채널명, 썸네일 URL, 마지막 갱신 시각
- **AC**:
  - [ ] 48시간 경과 시 자동 호출되어 채널 목록 업데이트
  - [ ] 신규 진입 채널은 `channels` 테이블에 추가 (활성)
  - [ ] 150위 밖으로 이탈한 채널은 비활성 처리 (즉시 삭제 X — 이력 보존)
  - [ ] 호출 실패 시 마지막 성공 목록을 계속 사용, 다음 주기에 재시도
  - [ ] 호출 결과를 화면에 직접 노출하지 않음 (백그라운드 후보)

#### F-2. YouTube Data API 구독자 수 폴링
- **주기**: 6시간마다 1회
- **대상**: 현재 활성 150개 채널의 구독자 수 (videos 수, view 수 등 포함 가능)
- **API**: `channels.list?part=statistics&id=...` (50개 묶음 호출 → 일 12회 호출 = 12 units/day)
- **AC**:
  - [ ] 6시간 경과 시 자동 호출 트리거
  - [ ] 응답을 `subscriber_snapshots`에 저장 (채널 ID, 시각, 구독자 수)
  - [ ] 구독자 수 기준으로 1~150위 재정렬
  - [ ] 상위 50개를 화면 표시 데이터로 결정 (또는 설정값 `DISPLAY_LIMIT`)
  - [ ] 호출 실패 시 마지막 성공 스냅샷 유지, 다음 주기에 재시도
  - [ ] API 쿼터 초과 시 백오프 (다음 주기까지 대기)

#### F-2b. 클라이언트 채널 SummaryCard 데이터 폴링
- **대상**: 환경변수 `CLIENT_CHANNEL_ID`로 지정된 단일 채널
- **갱신 주기**:
  - **시청 중 (라이브 시청자 수)**: 60초 (라이브 진행 중일 때만)
  - **좋아요**: 6시간 (다른 채널 폴링과 동일 사이클에 묶음)
- **AC**:
  - [ ] `CLIENT_CHANNEL_ID` 환경변수로 추적 채널 지정 (개발 중에는 임의의 채널)
  - [ ] 채널의 활성 라이브 스트림이 있는지 확인 (`search.list?eventType=live` 또는 `videos.list`)
  - [ ] 라이브 시청자 수를 60초마다 폴링하여 화면에 표시
  - [ ] 라이브가 없으면 "시청 중" 카드 표시값을 0으로 (또는 카드 숨김 — 구현 시 결정)
  - [ ] 좋아요 수는 6시간 폴링 사이클에 포함
  - [ ] 별도 API 쿼터 한도 추적 (라이브 폴링이 60초 주기라 쿼터 소비 큼)

### 4.2 화면 표시

#### F-3. 표시 범위 설정
- **AC**:
  - [ ] `DISPLAY_LIMIT` 상수(또는 환경변수)로 50/100 전환 가능
  - [ ] 50일 때 5개 컬럼 × 10행, 100일 때 10개 컬럼 × 10행 레이아웃
  - [ ] 두 모드 모두 1920×1080에서 가독성 유지
  - [ ] 백그라운드 데이터는 항상 150개 유지 (`BACKGROUND_LIMIT`)

#### F-4. 주기적 화면 갱신
- **AC**:
  - [ ] YouTube API 폴링 직후 화면이 새 데이터로 갱신 (push 또는 짧은 polling)
  - [ ] 갱신 시 4.3의 애니메이션 규칙을 따름
  - [ ] "업데이트" 카드의 상대 시간이 매초 갱신

### 4.3 애니메이션 (3종)

#### F-5. 순위 변동 (Position Swap)
- **트리거**: 폴링 결과, 기존 표시 채널 중 두 채널의 순위가 서로 바뀜
- **AC**:
  - [ ] 카드는 사라지지 않고 새 위치로 **부드럽게 이동** (FLIP 또는 transform 기반)
  - [ ] 이동 지속시간: 약 400~600ms, ease-out
  - [ ] 이동 중 카드에 짧은 강조 효과 (테두리/글로우, 약 800ms)
  - [ ] 콘텐츠(이름/숫자)는 이동 중에도 가독 가능

#### F-6. 50위권 진입 (Enter)
- **트리거**: 백그라운드(51~150) 채널이 표시 범위 안으로 진입
- **AC**:
  - [ ] 페이드 인 + 아래→위 슬라이드 (대안: 하이라이트 펄스)
  - [ ] 진입 직후 약 1.5~2초 동안 강조 상태 유지 (테두리/글로우)
  - [ ] 진입과 동시에 동일 위치에 있던 카드는 4.3.7 규칙으로 이탈

#### F-7. 50위권 이탈 (Exit)
- **트리거**: 표시 범위 안에 있던 채널이 50위 밖으로 밀림
- **AC**:
  - [ ] 페이드 아웃 + 아래로 밀려남 (대안: 흐려지며 사라짐)
  - [ ] 사라지는 동안 다른 카드의 재배치는 자연스럽게 이어짐
  - [ ] DOM에서 제거 (메모리 누수 방지)

#### F-8. 구독자 수 롤링 카운터
- **트리거**: 표시 채널의 구독자 수 변화 (실 폴링 또는 추정 보간)
- **AC**:
  - [ ] **자릿수별 순차 변화**: 1의 자리부터 시작, 왼쪽으로 자릿수별 순차 변경
  - [ ] 단순 숫자 보간(컴포넌트 통째 페이드/카운트업)이 아님
  - [ ] 디지털 카운터/전광판 느낌 (각 자리 숫자가 위/아래로 슬라이드)
  - [ ] 콤마(`,`) 유지, 자릿수 변화 시 레이아웃 흔들림 없음
  - [ ] 한 차례 롤링 완료까지 약 300~500ms (사용자가 읽을 수 있는 속도)
  - [ ] 증가 방향: 초록 색상 + 상승 화살표 / 감소 방향: 빨강 + 하락 화살표

### 4.4 추정 보간 (85% 알고리즘)

#### F-9. 구독자 수 부드러운 표시
실제 API 폴링은 6시간 주기지만, 화면에는 매 순간 숫자가 자연스럽게 증가하는 것처럼 보여야 한다.

- **입력**:
  - `S_prev`: 이전 폴링 시점 구독자 수
  - `S_curr`: 가장 최근 폴링 시점 구독자 수
  - `T_interval`: 폴링 간격 (6시간 = 21,600초)
  - `t`: 마지막 폴링 이후 경과 시간 (초)
- **계산**:
  - 평균 증가 속도: `r = (S_curr - S_prev) / T_interval` (초당 구독자 증가)
  - 다음 폴링 예측값: `S_predicted = S_curr + r × T_interval`
  - 안전 한계: `S_safe = S_curr + 0.85 × (S_predicted - S_curr)` (= 다음 폴링까지 도달할 안전한 최대값)
  - 화면 표시값:
    - `t < 0.85 × T_interval`: `S_curr + r × t` (정상 속도로 선형 증가)
    - `t ≥ 0.85 × T_interval`: 점진적 감속 (`S_safe`에 점근, 절대 추월 안 함)
- **AC**:
  - [ ] 화면 표시값은 **현재 알고 있는 가장 최근 API 값(`S_curr`)을 절대 추월하지 않는다** (다음 폴링이 도착하기 전까지의 보간값임)
  - [ ] 새 폴링 도착 시 표시값을 점진적으로 새 값에 맞춤 (점프 X, 약 1~2초 보정)
  - [ ] `S_curr - S_prev < 0` (감소)인 경우에도 동일 로직 적용 (감소 방향)
  - [ ] `S_prev` 데이터가 없는 첫 부팅 시: 보간 없이 `S_curr` 그대로 표시

> *수치 검증 예*: 브리프의 예시(API 1,000만 → 1,010만)가 정확히 위 공식으로 1,008.5만에서 감속하는지 단위 테스트로 검증.

### 4.5 순위 변동 임박 표시

#### F-10. 임박 카드 강조
- **트리거 (AND 조건)**: 인접 순위(N위, N+1위)의 두 채널이 **다음 두 조건을 모두** 만족할 때
  1. **절대값 조건**: 구독자 수 차이가 `RANK_ALERT_ABSOLUTE_THRESHOLD` 이하 (초기 제안: `10000`명)
  2. **증가율 조건**: 현재 증가 속도 차이로 계산했을 때 `RANK_ALERT_TIME_THRESHOLD_HOURS` 이내에 역전 가능 (초기 제안: `1`시간)
- **계산**:
  - 두 채널의 최근 폴링 기준 평균 증가율 차이 `Δr = r_lower - r_upper`
  - 역전 예상 시간 `t_flip = (S_upper - S_lower) / Δr` (단, `Δr > 0`인 경우만 의미 있음)
  - `t_flip < RANK_ALERT_TIME_THRESHOLD_HOURS` AND `(S_upper - S_lower) < RANK_ALERT_ABSOLUTE_THRESHOLD` → 임박
- **AC**:
  - [ ] 위 두 조건을 **모두 만족**할 때만 임박 판정
  - [ ] 인접한 **두 카드를 동시에** 강조 (한쪽만 X)
  - [ ] 카드 배경/테두리/은은한 빛(pulse 또는 glow) 효과
  - [ ] 과한 경고 문구/모달 X — 카드 강조만
  - [ ] RankAlertPanel의 숫자 배지가 임박 쌍의 개수를 반영
  - [ ] 임계값 빠져나오면 강조 해제
  - [ ] `Δr ≤ 0` (낮은 순위 채널이 더 느리게 증가)인 경우 임박 판정 X

### 4.6 장애 / 페일오버

#### F-11. API 호출 실패 처리
- **AC**:
  - [ ] Social Blade 또는 YouTube API 호출 실패 시 화면은 마지막 정상 데이터를 그대로 유지
  - [ ] LiveStatusPanel의 점/문구 색상만 경고색(주황 또는 노랑)으로 변경
  - [ ] 큰 에러 배너/모달/얼럿은 절대 표시 X (송출 안전)
  - [ ] 다음 주기에 자동 재시도 (별도 수동 개입 없음)
  - [ ] 호출 실패 이력은 서버 로그에 기록 (운영자가 사후 점검 가능)

## 5. 비기능 요구사항

### 5.1 인증
- 페이지 접근은 **HTTP Basic Auth** (ID/PW)
- 자격증명은 환경변수로 관리 (`BASIC_AUTH_USERNAME`, `BASIC_AUTH_PASSWORD`)
- OBS 브라우저 소스에서 URL에 자격증명 포함 가능 (`https://user:pw@host/`)

### 5.2 성능
- 1920×1080 해상도 / Chromium 최신 / OBS 30fps 캡처 환경 기준
- 60fps 애니메이션 권장 (rAF 기반), 최소 30fps 보장
- 단일 페이지 / 최대 150개 카드 (백그라운드 포함) — 성능 부담 낮음

### 5.3 보안
- **API 키 절대 커밋 금지**:
  - `.env` 파일 사용 / `.env.example`만 커밋
  - `자료/API키.txt` 파일은 `.gitignore`에 추가, 키는 `.env`로 이동
  - `자료/` 폴더 자체를 `.gitignore`에 추가하는 것을 권장 (커스터머 자료 보안)
- **HTTPS 미사용 / HTTP 평문 운영**:
  - 도메인 없이 Lightsail 고정 IP로만 접속
  - 운영자 1명 + 고정 IP 환경이라 HTTPS 설정 비용 대비 실익 낮다고 판단
  - 단, **Basic Auth 자격증명이 평문으로 전송됨**을 운영자에게 인지시키고 강력한 패스워드 사용 필수
  - 향후 HTTPS 필요 시 도메인 발급 → Let's Encrypt로 전환 가능 (별도 작업)

### 5.4 배포
- AWS Lightsail (Ubuntu, 도쿄 ap-northeast-1)
- Docker Compose로 한 번에 실행/중지
- 컨테이너 구성:
  - `app`: Next.js 애플리케이션
  - `worker`: 백그라운드 폴링 (별도 컨테이너 또는 app 내 cron — ARCHITECTURE에서 결정)
  - SQLite는 볼륨 마운트로 호스트 영속화

### 5.5 시간대
- 모든 표시 시각은 **JST** (`Asia/Tokyo`)
- 서버 내부 시각은 UTC, 표시 시점에 JST로 변환
- DB 저장 시각은 ISO 8601 UTC

**API 쿼터 추정 (YouTube Data API v3, 무료 한도 10,000 units/day)**:
- 메인 폴링 (150채널 × 4회/일, 50개 묶음): **12 units/일**
- 라이브 시청자 폴링 (60초): **약 1,440 units/일** (`videos.list`, 1 unit each)
- 라이브 스트림 탐지 (필요 시): `search.list` 1회당 100 units — 빈도 최소화 필요 (예: 1시간 1회)
- 합계: 약 1,500~2,000 units/일 — 한도의 20% 이내
